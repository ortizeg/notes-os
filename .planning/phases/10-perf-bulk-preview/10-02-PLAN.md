<objective>
Replace SortScreen's per-note by-id lazy preview load (0.80 s/note, with the dead
single-hop `_prefetch_next` chain) with a background bulk-load worker that pages bodies
from `get_inbox_note_bodies` into `_note_cache`. Threshold 250 / page size 200, first page
first. Below threshold: one silent bulk call, no indicator. At/above: paged load with a
non-blocking `Loading previews‚Ä¶ N/M` indicator that clears on completion. Keystrokes never
block (moves/skips only need `note.id` from the ref). Keep the by-id `get_note` path solely
as the cache-miss fallback for a note skipped faster than its page loads.

Purpose: This is the core skip-lag fix and the user-visible deliverable for PERF-01/02/03.
It consumes the bridge method from Plan 10-01.

Output: the bulk-load worker + page-merge logic + named threshold/page-size constants in
`sort.py`, the retired `_prefetch_next`, and Pilot tests for all three PERF behaviors.
</objective>

<interface_consumed>
From Plan 10-01 (Wave 1 ‚ÄĒ already merged when this runs):

    repo.get_inbox_note_bodies(offset: int, count: int) -> list[Note]

- Returns a contiguous page of fully-populated `Note` objects (id, title, body, preview)
  in folder order, id-aligned to `get_inbox_note_refs()`. Empty/out-of-range page ‚Üí `[]`.
- Available on `self.app.repo` (the BackingUpNotesRepository pass-through) and on
  `MockNotesRepository` (returns `self._inbox[offset:offset+count]`) for Pilot tests.
- The page at index `i` corresponds to `self._refs[offset + i]`; merge by `note.id`.
</interface_consumed>

<design_decisions>
Locked (from CONTEXT / SPEC ¬ß6 ‚ÄĒ do NOT re-litigate):
1. Module-level named constants (CLAUDE.md no-magic-numbers): `_BULK_THRESHOLD: int = 250`,
   `_BULK_PAGE_SIZE: int = 200`.
2. After `_apply_inbox_refs` populates `_refs`: if `len(refs) <= _BULK_THRESHOLD` ‚Üí one bulk
   call `get_inbox_note_bodies(0, len(refs))`, NO indicator. If `len(refs) > _BULK_THRESHOLD`
   ‚Üí load pages of `_BULK_PAGE_SIZE` starting at offset 0 (first page first), then subsequent
   pages; show `Loading previews‚Ä¶ N/M` where **N = notes loaded, M = total = len(refs)** ‚ÄĒ always
   a notes fraction, never mix pages and notes ‚ÄĒ and clear it on completion.
3. Each landed page merges into `_note_cache` keyed by `note.id`. Re-render the current note
   ONLY if its body just arrived (current `_index` falls within the page's `[offset, offset+len)`
   range ‚ÄĒ mirror the existing `_apply_note_body` stale-index guard). When re-rendering, resolve
   the body by `_note_cache[self._refs[self._index].id]` (id-keyed), NOT by positional page index
   ‚ÄĒ this self-corrects if AppleScript ever returns a page out of order (the phase's #1 named risk
   in SPEC ¬ßPhase 1).
4. The bulk load runs on a background `@work(thread=True)` worker; it must NEVER block a
   keystroke. `on_key` already works off `note.id` and is guarded only by `_loading` (refs).
   The bulk-body load does NOT set `_loading` ‚ÄĒ keystrokes stay live while bodies stream.
5. Retire `_prefetch_next` (delete the method and its call in `_apply_note_body`). Keep
   `_get_or_kick_note` / `_load_note_body` (by-id `get_note`) ONLY as the cache-miss fallback
   reached from `_render_current_note` when a note's body is not yet in `_note_cache`
   (skip-faster-than-load). `_apply_note_body` keeps its stale-index guard but no longer
   re-arms a prefetch chain.

The indicator lives in (or beside) the existing `#progress` Static. `_render_current_note`
already writes `Note {pos} of {total}` to `#progress`; the streaming indicator must not be
clobbered by that per-note render while pages are still loading. Track loading state in a
field (e.g. `_previews_loaded: int`, `_previews_total: int`, `_previews_streaming: bool`) and
compose the `#progress` text so the `Note X of Y` line and the `Loading previews‚Ä¶ N/M` line
coexist while streaming, and the previews line disappears on completion.
</design_decisions>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| background worker thread ‚Üí Textual event loop | The `@work(thread=True)` bulk worker marshals page results back via `call_from_thread`; it must never mutate UI state directly or block the keystroke path. |
| user keystroke ‚Üí router/move | Keystrokes act on `note.id` from refs while bodies stream; the move/skip path must stay live during body loading. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-05 | Denial of Service | bulk-load worker blocking the event loop | mitigate | Body load runs on `@work(thread=True)` and posts results via `call_from_thread`; `_loading` is NOT set during body streaming so `on_key` stays live. PERF-02 test (a) asserts a keystroke is honored mid-stream (T-06-07 preserved). |
| T-10-06 | Denial of Service | a failed/slow page aborting the session | mitigate | `_load_body_page` wraps the bridge call in `try/except Exception` (log + return); the by-id `get_note` cache-miss fallback (PERF-03) still resolves any note whose page failed ‚ÄĒ one bad page never strands the session. |
| T-10-07 | Tampering | stale/out-of-order page re-rendering the wrong note | mitigate | `_apply_body_page` re-renders only when `offset <= _index < offset+len(notes)` (range stale-guard, mirroring `_apply_note_body`'s `_index == index` guard); merges are id-keyed AND the current-note re-render resolves the body by `_note_cache[_refs[_index].id]` (id-keyed, not positional) so an out-of-order page self-corrects rather than showing the wrong note. |
| T-10-08 | Denial of Service | whole-inbox materialization on a huge inbox | accept | Bodies are paged (`_BULK_PAGE_SIZE`), not fetched whole; a 1000-note inbox materializes 200 at a time. Page count is bounded by `ceil(len(refs)/_BULK_PAGE_SIZE)`. |
| T-10-SC | Tampering | npm/pip/cargo installs | accept | No new packages this plan (Textual + stdlib already present); no Package Legitimacy Gate needed. |
</threat_model>
