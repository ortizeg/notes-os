behavior in `v3.0-speed-triage-ux-ROADMAP.md` › Phase 15 and `SPEC-speed-and-triage-ux.md`

`state.matches(...)` surface to call. `restore_counts` lives on `SortSession` (which already
owns the `moved`/`skipped`/`errors` counters) so the count math stays in one tested home.

save, None-on-missing/corrupt load, clear; path injectable, default

<design_decisions>
SETTLED HERE (documented in the module docstrings / model docstrings):

1. STALENESS = EXACT id-tuple equality. `SessionState.matches` returns True ONLY when
   `inbox_folder` is unchanged AND `note_ids` is exactly equal (same ids, same order) to the
   current refs' ids. This is the simplest correct rule: a reorder, an add, a remove, or a
   different inbox all flip it to stale. The Wave-2 screen treats stale state as "start over"
   and prefers NOT to prompt (it clears the stale file and renders at index 0) — so the user
   never sees a misleading "Resume at N" that points at the wrong note.

2. TIMESTAMP IS INJECTED, NOT DEFAULTED. `saved_at` is a required field with no default —
   the caller passes `datetime.now()` (production) or a fixed datetime (tests). A frozen
   Pydantic model must not bake `datetime.now()` into a `default_factory` for this state
   because determinism in tests and round-trip equality both depend on an explicit value.
   `saved_at` is persisted for diagnostics / future "resume a 3-day-old session?" logic; it
   is NOT part of the `matches` staleness check.

3. ATOMIC WRITE. `save` writes `model_dump_json()` to a temp file in the SAME directory
   (so `Path.replace` is an atomic same-filesystem rename) then `Path.replace`s it over the
   real path. A crash between the two leaves either the old file or no file — never a torn
   JSON. The parent directory is created (`mkdir(parents=True, exist_ok=True)`) before the
   temp write.

4. LOAD NEVER RAISES. `load` returns `None` when the file is missing, AND `None` when the
   file exists but is corrupt/unparseable (catching `OSError`, `ValueError`, and Pydantic's
   `ValidationError` — `model_validate_json` raises `ValidationError`, a subclass of
   `ValueError`, on malformed content). A corrupt file must degrade to "start over," never
   crash the launch (threat T-15-01).
</design_decisions>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| on-disk state file → ResumeStore.load | The JSON at `~/.notes-os/session-state.json` is untrusted input on each launch — it may be missing, truncated by a crash mid-write, hand-edited, or schema-invalid. `load` must never let bad bytes reach the screen as anything but `None`. |
| ResumeStore.save → the filesystem | A crash during the write must never corrupt the previously-good state file (atomicity). |
| SessionState.note_ids → staleness decision | The saved id signature drives whether 15-02 offers a resume; a wrong match would point "Resume at N" at the wrong note. |

| T-15-01 | Tampering | `ResumeStore.load` on corrupt/hand-edited JSON | mitigate | `load` catches `OSError`/`ValueError` (incl. Pydantic `ValidationError`) and returns `None` — a tampered or torn file degrades to "start over," never crashes the launch. Unit-tested with `"{ not json"` AND a schema-invalid JSON. |
| T-15-02 | Tampering | torn write from a crash mid-save | mitigate | `save` writes a temp file in the same directory then `Path.replace` (atomic same-fs rename) — a crash leaves the old file or none, never a half-written JSON.
