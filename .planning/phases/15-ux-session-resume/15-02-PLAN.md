<objective>
Wire user-facing session resume into `SortScreen`: save lightweight progress at each save
point, and on mount ALWAYS ASK "Resume at N / Start over" when a saved state still matches the
current inbox by note id (the locked behavior in `v3.0-speed-triage-ux-ROADMAP.md` › Phase 15
and `SPEC-speed-and-triage-ux.md` › Phase 6). This is the screen half of Phase 15; it consumes
the `SessionState` / `ResumeStore` / `restore_counts` / `ResumePromptModal` contract built in
15-01.

Purpose: deliver UX-03 end-to-end — quit at note 80 of 200 and pick up there next launch. A
materially changed inbox (ids no longer match) is detected and falls back to start-over safely
so the user never resumes onto the wrong note. The save is a fast local pathlib write on the
main thread (fine — bodies, not state, are what stream off-thread), and the resume decision
runs before the first render so it never fights the Phase-10 bulk-body worker (which fills the
id-keyed `_note_cache` independently).

Output: a `ResumeStore` DI seam on `SortScreen.__init__`, a `_save_session_state` helper wired
into `_advance` and `action_back`, the on-mount resume decision in `_apply_inbox_refs` that
pushes `ResumePromptModal` and applies Resume / Start over via the dismiss callback, a
clear-on-finish in `_complete_finish`, and Pilot tests proving every path (advance-saves,
resume-lands-with-restored-counts, start-over-clears, stale-no-prompt, finish-clears).
</objective>

<save_points_and_resume_contract>
SETTLED HERE (documented in the SortScreen docstrings):

SAVE POINTS (when `_save_session_state` runs):
1. At the END of `_advance`, for the JUST-processed note — but ONLY when the session is still
   live (NOT when `_advance` reached the end and finished: `_index < len(self._refs)`). A
   finished/empty session has nothing to resume and must not leave a stale file.
2. In `action_back` at `AWAIT_CATEGORY` (leaving mid-session via pop_screen), BEFORE
   `self.app.pop_screen()` — but ONLY if the user has actually started
   (`0 < self._index < len(self._refs)`; saving an index-0 unstarted session is pointless and
   would be filtered as not-resumable on reload anyway).

NEVER SAVE when `_inbox_empty` is True (empty inbox or post-finish summary state) or when
`_index >= len(self._refs)`.

RESUMABILITY (computed in `_apply_inbox_refs` after `_refs` is set, BEFORE the first render):
`state = self._store.load()`. The state is RESUMABLE iff:
  - `state is not None`, AND
  - `state.matches(inbox_folder, current_ids)` where `inbox_folder =
    app.app_config.bridge.inbox_folder` and `current_ids = tuple(r.id for r in self._refs)`
    (the 15-01 exact id-tuple + inbox equality rule), AND
  - `0 < state.index < len(self._refs)` (a saved index of 0 or out-of-range is not a useful
    resume point).
If RESUMABLE → ALWAYS push `ResumePromptModal(index=state.index, total=len(self._refs))` and
defer the first render to the dismiss callback (always-ask). If NOT resumable → if a (stale)
file exists, `self._store.clear()`; render at index 0 with NO prompt.

DISMISS CALLBACK (`_on_resume_decision(resume: bool | None)`):
  - `resume is True` → `self._index = state.index`;
    `self._session.restore_counts(state.moved, state.skipped, state.errors)`; then
    `self._render_current_note()` and (re)start the bulk body load if not already running.
  - else (False / None) → `self._store.clear()`; `self._index = 0`;
    `self._render_current_note()`.
Capture `state` into the callback closure (or store it on `self` transiently) so the callback
has the saved index/counts.

WORKER COOPERATION (Phase-10 no-race): the bulk body load is started in `_apply_inbox_refs`
today via `_start_bulk_body_load()` right after the first render. With resume, start the bulk
load UNCONDITIONALLY after the resume decision is SET IN MOTION (it fills the id-keyed
`_note_cache` regardless of `_index`); the resume callback only sets `_index` and re-renders,
which is an id-keyed lookup — a page landing before or after the modal is dismissed resolves
the correct note by id (the same self-correcting guard Phase-10 already relies on). Do NOT
gate the bulk load behind the modal; do NOT restart it in the callback if it is already
running. The clean approach: in `_apply_inbox_refs`, if resumable, push the modal and return
WITHOUT rendering/starting bulk load; render + start bulk load happen in the callback for BOTH
branches. If not resumable, render at 0 + start bulk load inline as today. (Either ordering is
race-free since the cache is id-keyed; pick the modal-defers-render ordering for clarity.)

CLEAR POINTS: `_complete_finish` (session done → `self._store.clear()`); the Start-over branch
of the callback; and the not-resumable stale-file branch in `_apply_inbox_refs`.
</save_points_and_resume_contract>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| on-disk saved state → SortScreen on mount | The state loaded at launch is untrusted; even after 15-01's None-safe load, a well-formed but STALE state (inbox changed) must not be allowed to point the user at the wrong note. |
| _save_session_state → the filesystem (main thread) | A state write happens on the event-loop thread at each save point; a filesystem failure there must not crash an in-progress triage session. |
| ResumePromptModal dismiss → _index / counts | The modal result drives a jump of `_index` and a count restore; a wrong index would resume onto the wrong note or past the end. |

## STRIDE Threat Register
