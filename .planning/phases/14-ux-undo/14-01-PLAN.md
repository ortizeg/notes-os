<objective>
Add a session-owned, LIFO undo stack to `SortSession` so the most-recent skip or move
can be reversed, repeatably, unbounded within the session (the locked behavior in
`v3.0-speed-triage-ux-ROADMAP.md` › Phase 14 and `SPEC-speed-and-triage-ux.md` › Phase 5).
This plan delivers ONLY the UI-agnostic session layer (a clean contract the Wave-2 screen
plan consumes); no screen wiring and no writes happen here.

Purpose: `SortSession` already owns the event history (`_events`) and the
`moved`/`skipped`/`errors` counters and the `record_move_failure` reconciliation, so it is
the correct home for the undo stack and the counter-reversal logic (design directive 1).
Putting the reversal here keeps the count math in one tested place and gives the screen a
small, pure `pop_undo()` contract to call.

Output: a frozen `UndoEntry` value object, a `SortSession._undo_stack`, pushes wired into
`record_move`/`record_skip`, a `pop_undo() -> UndoEntry | None` that reverses counters
LIFO, the errored-event rule wired into `record_error`/`record_move_failure`, and unit
tests holding the session module at its ≥95% coverage floor.
</objective>

<errored_event_rule>
CHOSEN RULE (documented in the `pop_undo` / `record_*` docstrings):

Only a SUCCESSFUL `record_move` or `record_skip` pushes an undo entry. `record_error`
pushes NOTHING (an errored move never counted as `moved` and the note never left the inbox,
so there is nothing to reverse). `record_move_failure` (the optimistic-move reconciliation)
pushes NOTHING and additionally REMOVES the undo entry it reconciles: when an optimistic
move is later found to have failed off-thread, its already-pushed undo entry must be popped
out of the stack (matched by `note_id` + `kind == "move"`, most-recent first) so the user
can never "undo" a move that did not actually happen. This keeps the stack in one-to-one
correspondence with reversible state and guarantees `pop_undo` never tries to move a note
back that is already sitting in the inbox.
</errored_event_rule>

## Trust Boundaries

|----------|-------------|
| screen → SortSession | The screen calls `record_move`/`record_skip`/`pop_undo`; SortSession owns the counters + stack and must never let them desync or go negative. |
| SortSession counters → SessionSummary / audit log | Reversed counts must flow into `summary()` and `write_log()` so the end-of-session report is truthful after undos. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-01 | Tampering | `pop_undo` counter math | mitigate | Guard each decrement with `> 0` (mirror the `record_move_failure` guard) so a stray/duplicate `pop_undo` can never drive `moved`/`skipped` negative; unit-tested by the guard case in Task 1. |
| T-14-02 | Tampering | undo stack vs. reconciled failures | mitigate | `record_move_failure` removes the reconciled move's `UndoEntry` so a move that never landed can never be undone (errored-event rule); unit-tested (`move_failure pops the reconciled entry`). |
| T-14-03 | Repudiation | audit log fidelity | accept | The original MOVE/SKIP `_NoteEvent` line is kept append-only; the undo's own move-back is logged by the screen when it records it. An undo does not erase history — by design. |
| T-14-04 | Denial of Service | unbounded undo stack | accept | The stack grows at most one entry per processed note (≤ inbox size, already materialized in `_refs`); LIFO-to-session-start is the locked requirement (no cap). Memory cost is bounded by the session note count — no new unbounded growth. |
| T-14-SC | Tampering | npm/pip/cargo installs | mitigate | No new packages added in this plan; nothing to audit. (No `## Package Legitimacy Audit` needed — zero installs.) |
