<objective>
Wire the user-facing Undo (`U`) into `SortScreen`: a `U`/`u` binding that pops the
most-recent `UndoEntry` from `SortSession` (built in 14-01), reverses it on screen, and —
for a moved note — issues the move-back through the EXISTING Phase-13 serialized off-thread
write queue. This is the screen half of Phase 14 (the locked behavior in
`v3.0-speed-triage-ux-ROADMAP.md` › Phase 14 and `SPEC` › Phase 5).

Purpose: deliver UX-02 end-to-end — confidence → speed. `U` undoes the last skip or move,
repeatably, LIFO to session start. A skip-undo is pure index navigation (no write); a
move-undo moves the note back to its captured source folder. Reusing the Phase-13 queue
(never a synchronous on-event-loop write) preserves the single-writer backup-latch safety
and move ordering invariants.

Output: a `U`/`u` binding + `_handle_undo`, source-path capture at move time, move-back
enqueued via `_enqueue_write`, skip-undo index step, an empty-stack notify hint, and Pilot
tests (move→undo, skip→undo, repeatable-to-start, empty-stack no-op, undo-failure surfaced)
that reuse the Phase-13 worker-wait + `threading.Event` gating discipline for determinism.
</objective>

<interaction_with_inflight_writes>
Moves are optimistic and serialized FIFO. An undo issued right after a move enqueues the
move-back AFTER the original move in the SAME queue → the note is moved to the destination
then moved back to the inbox (correct end state). Do NOT attempt to cancel the in-flight
move; just enqueue the reversal. This is the exact reuse the Phase-13 code anticipated
(`_enqueue_write` docstring: "Reusable by Phase 14 (Undo) for move-back writes").
</interaction_with_inflight_writes>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| keystroke (`U`) → SortScreen | An untrusted keystroke triggers a write (move-back). Must be a no-op in every state where there is nothing to undo (loading, inbox-empty, finished, empty stack, mid-folder-nav). |
| SortScreen → Phase-13 write queue | The move-back is a write; it must go through the single serialized drainer (backup-latch + FIFO safety), never a synchronous on-event-loop write. |
| off-thread move-back → counters/UI | A failed move-back must not crash the TUI, drop the note, or drive counts negative. |
