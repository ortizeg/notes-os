<objective>
Replace the literal `Loading preview…` text shown while a note body is still loading with a dim,
CSS-class-driven skeleton placeholder so a pending preview reads as "coming," not "stuck" (UX-01).

The single render path that decides preview content — `SortScreen._render_current_note` — is the
only behavioral seam. When `_current_note is None` (body not yet cached) it must render a muted
multi-line skeleton placeholder into `#note-preview` AND apply a skeleton CSS class to that Static;
when the body is cached it must render the real preview AND remove the class so a loaded note never
keeps the dim style. The skeleton text comes from a module-level named constant (no magic string,
per CLAUDE.md) and the dim appearance is entirely CSS-driven via `app.tcss`.

Purpose: a pending preview communicates progress, not a hang — the last polish on the Phase-10
background bulk-load UX. Pure UI: no bridge, no router, no write-path change.
Output: a skeleton constant + skeleton CSS-class name in `sort.py`, a dim style in `app.tcss`, an
updated render seam, and Pilot coverage for skeleton-then-real.
</objective>
