<objective>
Give the `Router` a resolve-only move seam and give `SortSession` a way to convert an
optimistically-counted move into an error — the two non-UI primitives that Plan 13-02's
optimistic-advance + off-thread-write SortScreen depends on.

1. Router seam (PERF-04 enabler): add a `defer_writes: bool = False` constructor flag.
   When `True`, the shared `_do_move` helper RESOLVES the terminal move — it builds and
   returns a `RouteResult(state=SHOW_NOTE, action=MOVE, folder_path=..., display_path=...)`
   — but SKIPS the synchronous `ensure_folder` + `move_note` calls. When `False` (the
   default, used by the CLI `SortController`), behaviour is byte-identical to today: the
   router writes synchronously inside `_do_move`. SortScreen (13-02) constructs its Router
   with `defer_writes=True` and dispatches the actual write off-thread itself.

2. Session count-conversion (PERF-05 enabler): add `SortSession.record_move_failure(note_id)`.
   Because 13-02 records the move OPTIMISTICALLY (before the off-thread write runs), a later
   write failure must turn that move back into an error so the session summary counts stay
   correct (`moved` ⇒ `errors`). This belongs in the session model, not the screen.

Purpose: isolate the two pure, deterministically-testable primitives so the concurrency-
sensitive screen work in 13-02 is built on a unit-proven foundation, and the CLI write path
is provably untouched.
Output: `Router.defer_writes` + a resolve-only `_do_move` branch; `SortSession.record_move_failure`;
router unit tests proving both deferred-resolve and default-write behaviour (>=95% on
router.py write path); session unit tests for the count conversion.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Router caller → `_do_move` write side-effect | Today every terminal selection writes to the 92 MB Notes DB synchronously inside the router. The `defer_writes` flag relocates that write to the caller (SortScreen, 13-02). The CLI caller MUST keep the synchronous write — a mis-set default would silently stop the CLI from moving notes. |
| Optimistic count → session summary | A move is counted before its off-thread write is known to succeed (13-02). `record_move_failure` is the sole path that re-balances the count when the write fails — a missing/incorrect conversion would over-report moves and lose an error. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-01 | Tampering / Data integrity | `Router._do_move` deferred branch | mitigate | The deferred branch returns the IDENTICAL `RouteResult` (same `folder_path`/`display_path`) so the caller writes to the SAME resolved path the router computed — no path drift between resolve and write. Covered by `test_defer_*_resolves_without_write` asserting the exact resolved path. |
| T-13-02 | Denial of service / Silent breakage | CLI `SortController` write path | mitigate | `defer_writes` defaults `False`; `build_default_controller` is asserted UNCHANGED so the CLI still writes synchronously. Covered by the default-write router tests AND (in 13-02's plan) the controller integration suite staying green. |
| T-13-03 | Repudiation / Count loss | optimistic move never reconciled on failure | mitigate | `record_move_failure` decrements `moved` and increments `errors` and rewrites the audit event to `[ERROR]`, so the session log truthfully records the note stayed in the inbox. Covered by the session count-conversion tests. |
| T-13-04 | Data integrity (negative count) | `record_move_failure` with no prior move | mitigate | The method guards `moved > 0` before decrementing and falls back to a fresh error when no matching move event exists — counts can never go negative. Covered by the no-prior-move defensive test. |
| T-13-SC | Tampering | npm/pip/cargo installs | n/a | No new dependencies introduced this plan (no package installs); the package-legitimacy gate does not apply. |
</threat_model>
