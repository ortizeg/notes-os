<objective>
Make a move feel as instant as a skip. Today the move (archive [X] and folder/subfolder
selection) runs `ensure_folder` + `move_note` + the session's first-write 92 MB backup
SYNCHRONOUSLY inside the Router on the Textual event loop — freezing the UI ~1 s on the
first move and for every osascript move. This plan moves that write OFF the event loop.

Behaviour after this plan (using the `Router.defer_writes` seam + `SortSession.record_move_failure`
from 13-01):

1. SortScreen constructs its Router with `defer_writes=True` (so the router RESOLVES the move —
   `folder_path` + `display_path` — but no longer writes). Navigation transitions
   (AWAIT_FOLDER / AWAIT_SUBFOLDER, which only do the fast `get_para_structure` read) stay
   synchronous — only the terminal write is deferred.

2. Optimistic advance: when a `handle_*` call returns `action == MOVE`, CAPTURE `note.id` +
   `folder_path` + `display_path` into locals FIRST (no advance/worker race), then
   `_session.record_move(...)`, set `app.sort_in_progress = True`, show a non-blocking
   `moved ✓ → {display}` confirmation, run `_after_move` (extraction OFF by default — current
   behaviour preserved), and `_advance()` IMMEDIATELY. All synchronous, NO AppleScript.

3. Serialized off-thread write: each pending move `(note_id, folder_path, display)` is enqueued
   on a FIFO `collections.deque` and drained by a SINGLE in-flight `@work(thread=True)` worker.
   Exactly one writer runs at a time → the Phase-12 backup latch (`_session_backed_up` /
   `_maybe_backup_once`, NOT thread-safe) and osascript move ordering stay safe. `begin_session()`
   stays on the main thread in `_apply_inbox_refs` (unchanged from Phase 12).

4. Failure surfacing (PERF-05, never silently drop): a queued write that raises
   NotesOSError/BackupError marshals back to the main thread via `call_from_thread`, calls
   `_session.record_move_failure(note_id, msg)` (converts the optimistic move into an error),
   shows a non-blocking `self.notify(...)` naming the note that stayed in the inbox, and appends
   to a `_failed_moves` list. The note physically stayed in the inbox (the move never ran) — it is
   NOT lost. T-06-05 (one bad move never aborts the session) is preserved.

5. Quit/finish reconciliation: `_finish` DRAINS pending writes before finalizing the summary so
   landed failures are reflected in the counts (drain-before-summary). While writes are in flight
   `app.sort_in_progress` stays True so the ConfirmQuitModal guard still fires on Q.

Purpose: eliminate the per-move UI freeze (PERF-04) while never losing a note and keeping counts
correct on failure (PERF-05).
Output: a `defer_writes=True` Router, an optimistic `_handle_move`, a serialized `_write_queue` +
`_drain_write_queue` worker (named reusably for Phases 14/15 move-back/resume), a `_failed_moves`
list, drain-on-finish reconciliation, and Pilot tests proving never-freeze + failure handling +
count correctness.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Event-loop thread (UI) ↔ write-drainer worker thread | The drainer runs `ensure_folder`/`move_note` (and thus the Phase-12 backup latch + osascript) off the event loop. All UI/state mutation MUST marshal back via `call_from_thread`; the worker only touches `app.repo` and the FIFO queue. |
| Optimistic count (UI) ↔ eventual write result (worker) | A move is counted before its write succeeds. The worker's success/failure must reconcile the count via `_on_write_failed` → `record_move_failure`, on the main thread. |
| User keystroke (advance) ↔ pending write (note.id) | The note id + folder_path are captured into locals BEFORE `_advance` so a later index change cannot mis-target the in-flight write. |
